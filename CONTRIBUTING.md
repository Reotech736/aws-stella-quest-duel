# CONTRIBUTING.md

このリポジトリへの変更は、小さく・安全に・意図が追いやすい形で進めます。

## 基本方針

- まず設計と仕様整理を優先する
- 大きな変更は小さな単位に分ける
- 不要な依存関係を増やさない
- 実AWSリソースへの変更やデプロイは、事前確認なしに行わない
- 秘密情報をコミットしない

## ブランチ戦略

当面は、軽量な運用を優先して `main` + 短命ブランチの形を採用します。

- `main`: 常に最新の基準ブランチ
- 作業時は `main` からブランチを切る
- 作業ブランチは短命に保ち、1つの目的に絞る
- 大きな機能でも、できるだけ分割してマージする

ブランチ名の例:

- `docs/update-readme`
- `docs/add-rules-draft`
- `feature/mvp-room-api`
- `feature/game-state-model`
- `fix/turn-validation`
- `chore/update-ignore`

この方針を推す理由:

- まだ初期段階で、`develop` などの恒久ブランチを増やす運用コストに見合いにくい
- 学習用かつMVP前なので、ルールは少ない方が維持しやすい
- 変更単位を小さくするとレビューや差分確認がしやすい

将来、並行開発が増えたら `release/*` や `hotfix/*` の導入を検討します。

## コミットメッセージ

コミットメッセージの先頭には、変更種別を示す接頭辞を付けます。

形式:

`<Type>: <summary>`

例:

- `Add: plan.md の初版を追加`
- `Docs: README に開発方針を追記`
- `Fix: ターン更新時のバリデーション漏れを修正`
- `Refactor: game state の型定義を整理`
- `Test: action validator のテストを追加`
- `Chore: .gitignore を更新`

推奨する接頭辞:

- `Add:` 新規追加
- `Docs:` ドキュメント追加・更新
- `Fix:` バグ修正
- `Refactor:` 振る舞いを変えない整理
- `Test:` テスト追加・更新
- `Chore:` 雑多な保守作業、設定変更

ルール:

- 1コミット1目的を意識する
- 何をしたかが読める短い要約にする
- 複数種類の変更が混ざる場合でも、主目的に合わせて接頭辞を選ぶ

## 変更前の確認

変更前に、以下を確認します。

- 関連する `AGENTS.md`
- 既存の設定ファイル
- ロックファイルや使用ツール
- 変更影響範囲

必要なら以下のような確認を先に行います。

現在の状態を確認するコマンドです。

```bash
git status --short
```

差分を確認するコマンドです。

```bash
git diff
```

## 変更内容の考え方

- まずは設計や仕様の文章化から進める
- 実装はMVPに必要な範囲に限定する
- ゲームルール判定はサーバー側に置く
- クライアント側は表示と入力を中心に保つ

## マージ前チェック

少なくとも以下を確認します。

- 変更意図が説明できる
- 不要な依存関係が増えていない
- 秘密情報が含まれていない
- 差分が追いやすい
- 必要なら動作確認方法が書かれている

## AWSに関する注意

- AWS CLI や IaC での実リソース変更は、明示的な確認がある場合のみ行う
- 本番系や共有環境へ影響しうる操作は、対象と影響範囲を先に整理する
- 学習目的であっても、破壊的変更は慎重に扱う
