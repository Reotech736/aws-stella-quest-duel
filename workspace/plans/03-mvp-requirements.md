# 03-mvp-requirements

## 概要

このドキュメントは、「ステラクエスト Duel」をオンラインで1ゲーム開始し、終了まで進行できる最小構成を定義するためのMVP要件です。

ゲームルールは `workspace/plans/02-rules-discovery.md` を正とし、この文書ではユーザー体験、必要機能、対象外機能、完了条件を整理します。

## MVPのゴール

認証済みの人間プレイヤー2人が同じルームへ参加し、ルーム作成者の開始操作によってゲームを初期化し、HTTP APIとポーリングを使ってDuelルールの対戦を最後まで完了できることをMVPのゴールとします。

## 公開ポリシー

状態: 確定

初期MVPは、管理者が招待した少人数の知人だけが利用できる限定ベータとして公開します。

- Amazon Cognitoの自己登録を無効化する
- 管理者がAWSコンソールから利用者アカウントを手動作成する
- 匿名利用は許可しない
- 認証済みユーザーであれば誰でもルームを作成できる
- 公開ルーム一覧やルーム検索は提供しない
- Amplify Hostingは限定試験中、ブランチ単位の共有パスワードで保護する
- Amplifyの共有パスワードを解除する場合は、公開範囲とリスクを再検討して明示的に決定する
- API Gateway HTTP APIではCognito JWT認証を必須とする

Amplifyの共有パスワードはアプリ入口の保護、Cognitoは利用者ごとの認証、ルームIDは認証済みユーザーが特定ルームへ参加するためのキーとして、それぞれ別の目的で使用します。

## 対象ユーザー

- ルーム作成者
  - 対戦ルームを作成する
  - 相手プレイヤーの参加を待つ
  - スタートプレイヤーの決定方法を選んでゲームを開始する
- ルーム参加者
  - 作成済みルームへ参加する
  - ルーム作成者がゲームを開始するまで待つ
- ダミー
  - ユーザーではなく、ゲームルールに従ってサーバーが自動操作する第3参加者

## 最小ユーザーフロー

1. プレイヤーが認証する
2. 一方のプレイヤーが対戦ルームを作成する
3. もう一方のプレイヤーが同じルームへ参加する
4. ルーム作成者が開始方法を選ぶ
5. サーバーがスタートプレイヤーを確定する
6. サーバーがカードをシャッフルし、配札とゲーム状態の初期化を行う
7. 両プレイヤーがポーリングで最新のゲーム状態を取得する
8. 各プレイヤーが自分の手番に許可された操作を行う
9. サーバーがダミーのプレイ、ラウンド判定、ペナルティ、勝敗判定を行う
10. 悟りまたは光の喪失によってゲームが終了する
11. 両プレイヤーが勝者、敗者、終了理由を確認する

## 機能要件

### 1. 認証

状態: 確定

- Amazon Cognitoを利用してユーザーを識別する
- Cognitoの自己登録を無効化する
- 管理者がAWSコンソールから利用者アカウントを手動作成する
- 未認証ユーザーはルーム作成、ルーム参加、ゲーム操作を行えない
- APIは認証済みユーザーの識別子を信頼できる認証情報から取得する
- クライアントが送信した任意の `userId` だけを操作主体として信用しない

MVPでは、自己登録画面、利用申請画面、利用者管理画面を実装しません。初回ログインやパスワード再設定の具体的な画面フローは、Cognitoの設定と画面設計で決定します。

### 2. ルーム作成

状態: 確定

- 認証済みユーザーがルームを作成できる
- 作成者はルーム内のプレイヤーAとして記録する
- 作成直後のルームは参加者待ち状態とする
- 1ルームに参加できる人間プレイヤーは2人までとする
- 作成者自身が参加者枠へ重複参加できない
- 待機中または進行中のルームへ所属しているユーザーは、別のルームを同時に作成できない
- 参加者待ちルームは作成から24時間で失効する
- CPUプレイヤーを選択する機能は提供しない

### 3. ルーム参加

状態: 確定

- 認証済みユーザーが参加者待ちのルームへ参加できる
- 参加者はルーム内のプレイヤーBとして記録する
- 参加には6桁のルームIDを使用する
- ルームIDの文字集合は `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` とする
- 入力されたルームIDは大文字へ正規化して検証する
- ルームIDは暗号学的に安全な乱数で生成する
- 有効なルームIDと衝突した場合は、サーバーが再生成する
- すでに2人揃っているルームには参加できない
- 作成から24時間を過ぎたルームには参加できない
- ゲーム開始後に新しいプレイヤーは参加できない
- ルーム作成者自身は参加者枠へ参加できない
- 待機中または進行中のルームへ所属しているユーザーは、別のルームへ同時に参加できない
- 存在しない、期限切れ、満員、開始済みのルームIDに対しては、理由を区別せず「参加できませんでした」と表示する

ルームIDは認証の代替ではありません。Cognitoで認証済みのユーザーだけが、ルーム参加APIへルームIDを送信できます。

### 4. ルーム参加試行の制限

状態: 確定

- ユーザー単位で失敗した参加試行を記録する
- 15分間に5回失敗した場合、その後15分間は新しい参加試行を拒否する
- 制限中のユーザーには、個別のルーム状態を明らかにしない共通エラーを返す
- API Gatewayにもルート別スロットリングを設定する
- 具体的なスロットリング値は、正規利用のポーリングと操作を妨げない範囲でAPI設計時に決定する

### 5. ゲーム開始

状態: 確定

- 人間プレイヤーが2人揃った後だけ開始できる
- 開始操作はルーム作成者だけが実行できる
- 作成者は次の開始方法から1つを選ぶ
  - ランダム
  - ルーム作成者を先手にする
  - ルーム参加者を先手にする
- サーバーは最初にスタートプレイヤーを確定する
- スタートプレイヤー確定後にカードをシャッフルして配札する
- 同じルームから複数のゲームを同時に開始できない
- 開始操作の重複送信で再シャッフルや再配札を発生させない

### 6. ゲーム状態の表示

状態: 確定

両プレイヤーに、少なくとも次の公開情報を表示します。

- 両プレイヤーの表示名または識別可能な名前
- 現在のスタートプレイヤー
- 現在操作すべきプレイヤー
- 現在のフェーズ
- 両プレイヤーの手札枚数
- 両プレイヤーの手札の色
- 両プレイヤーの光面と闇面の枚数
- 黒い星マーカーの位置
- デッキの残り枚数
- デッキトップの色
- 捨て札トップのカード
- 現在のラウンドでプレイされたカード
- 両プレイヤーの収集カード
- ゲームの進行状態と終了結果

本人には、自分の手札について数字を含む完全な情報を表示します。

相手には、相手手札の数字、デッキトップの数字、デッキ2枚目以降の色と数字を表示しません。

### 7. プレイヤー操作

状態: 確定

ルール上必要な次の操作を提供します。

- 光面の星明りトークンを1枚消費して、手札上限まで最大3枚引く
- 手札からカードを1枚プレイする
- カードプレイ後に手番を終了する
- ラウンド勝者が収集する感情カードを選ぶ
- 必要な場合、ラウンド勝者が次の捨て札トップを選ぶ

クライアントは操作可能な選択肢を分かるように表示しますが、最終的な実行可否はサーバーが判定します。

### 8. サーバーによる自動処理

状態: 確定

- 空になった手札の補充
- ダミーによるデッキトップのプレイ
- デッキ枯渇時の再構築
- リード、トランプ、スーパートランプの判定
- ラウンド勝者の決定
- 連勝および重複収集による光喪失
- 悟りと敗北の判定
- 黒い星マーカーの移動
- 次ラウンドのスタートプレイヤー決定
- ゲーム終了状態の保存

### 9. 状態同期

状態: 確定

- クライアントはHTTP APIでゲーム状態を取得する
- 相手の操作結果は定期ポーリングで反映する
- WebSocketとAppSync SubscriptionはMVPで使用しない
- 相手ターンまたは相手の選択待ちは2秒間隔でポーリングする
- 参加者待ち画面は5秒間隔でポーリングする
- 自分が操作可能な手番は10秒間隔でポーリングする
- ブラウザタブが非表示の場合は30秒間隔へ延長する
- ブラウザタブが表示状態へ戻った場合は即座に最新状態を取得する
- 操作成功後は、可能な限りAPIレスポンスに含まれる最新状態を反映する
- 操作レスポンスに最新状態を含められない場合は、操作成功直後に状態を再取得する
- ゲーム終了後はポーリングを停止する
- ブラウザを再読み込みしても、認証済みの参加者は進行中のゲーム状態を再取得できる
- サーバーは状態の更新競合を検出し、古い状態に基づく不正な操作を拒否する

### 10. 操作履歴

状態: 確定

- サーバーで受理した主要操作をイベントログとして保存する
- 少なくとも操作者、操作種別、対象ゲーム、発生日時を追跡できるようにする
- 秘密情報や、相手に公開してはいけない手札情報をログへ不用意に出力しない
- MVPではユーザー向けの対局履歴閲覧画面を必須としない
- 終了済みゲーム状態とイベントログは、終了から30日後にDynamoDB TTLで自動削除する

### 11. ゲーム終了

状態: 確定

- 悟りによる勝利を表示する
- 相手がすべての光を失ったことによる勝利を表示する
- 勝者、敗者、終了理由を両プレイヤーへ表示する
- 終了後のゲーム操作を拒否する
- 終了済みゲームの最終状態を再取得できる

## 非機能要件

### セキュリティと情報秘匿

- Amplify Hostingの限定試験用ブランチを共有パスワードで保護する
- すべてのゲーム操作で認証とルーム参加権限を検証する
- ゲームルールの判定をクライアントへ委ねない
- 閲覧者ごとにレスポンスを組み立て、非公開情報を返さない
- デッキの完全な並び順をクライアントへ送らない
- ルームIDからルームの存在や状態を列挙できないエラー表現にする
- APIキー、トークン、パスワード、個人情報をリポジトリやログへ保存しない

### 整合性

- 1つの操作を原子的に検証・更新する
- 重複リクエストでカードの二重ドローやトークンの二重消費を起こさない
- 同時操作では、先に成立した更新だけを採用し、競合した操作を明示的に拒否する
- ゲーム状態とイベントログの関係を追跡できるようにする

### 運用

- AWS Lambdaの処理結果とエラーをCloudWatch Logsで確認できる
- 認証情報や手札・デッキの完全情報を通常ログへ出力しない
- CloudWatch Logsの保持期間を14日とする
- 月額5 USDのAWS Budgetを作成し、50%、80%、100%到達時に通知する
- AWS Budgetは費用を自動停止する上限ではないため、通知後は利用状況を手動確認する
- 初期MVPではAWS WAF、NAT Gateway、VPC接続、カスタムドメイン、Lambda Provisioned Concurrencyを使用しない
- MVPでは小規模利用を前提とし、自動スケーリングの細かな最適化は後回しにする

## MVP対象外

### 恒久的に実装しない

- CPU対戦

### MVP後に必要性を再検討する

- 観戦
- ランキング
- チャット
- 課金
- WebSocket
- AppSync Subscription
- 対局履歴の閲覧画面
- リプレイ
- 複数人戦
- 複数ゲーム形式や別ルール

### MVPでは実装しない

- AIによる手の推薦
- ゲーム内のボイス通話
- フレンド管理
- 大会機能
- 高度な不正検知

## MVP完了条件

- 認証済みユーザーがルームを作成できる
- 自己登録できず、管理者が作成したユーザーだけが利用できる
- 6桁のルームIDを使って別の認証済みユーザーがルームへ参加できる
- 待機中または進行中のルームへ同じユーザーが重複所属できない
- 無効なルームIDの連続試行がユーザー単位で制限される
- ルーム作成者がスタートプレイヤーの決定方法を選んで開始できる
- サーバーが正しい順序でシャッフル、配札、初期化を行う
- 両プレイヤーが公開範囲の異なるゲーム状態を取得できる
- ルール上必要なすべてのプレイヤー操作を実行できる
- ダミーとラウンド解決をサーバーが自動処理できる
- ブラウザ再読み込み後も進行中のゲームへ復帰できる
- 悟りまたは光の喪失までゲームを完了できる
- 終了後に勝敗と終了理由を確認できる
- 非公開情報が相手プレイヤーへ漏れない
- 重複送信や同時更新でゲーム状態が二重更新されない
- CPU対戦やMVP対象外機能が含まれていない

## 要決定事項

以下はゲームルールではなく、MVPのユーザー体験として決定が必要です。

1. ルーム作成者が開始前に退出した場合の扱い
2. 参加者が開始前に退出した場合、同じ枠へ別ユーザーが参加できるか
3. ゲーム中に切断したプレイヤーの扱い
   - 無期限に再接続を待つ
   - 一定時間後に敗北とする
4. 投了機能をMVPへ含めるか
5. MVPで対象とするブラウザと画面サイズ

## 次の設計へ渡す事項

### データ設計

- ユーザー、ルーム、ゲーム、プレイヤー、ゲーム状態、イベントログの保存単位
- 6桁ルームIDの一意性確認と24時間の失効管理
- ユーザーが所属できる待機中・進行中ルームを1つに制限する方法
- ユーザー単位の参加失敗回数と制限解除時刻
- 30日後に削除するDynamoDB TTL属性
- 閲覧者ごとの公開状態を生成するために必要なカード情報
- 状態更新の競合を検出するバージョン情報
- 未開始、進行中、終了済みを表す状態

### API設計

- 認証、ルーム作成、ルーム参加、ゲーム開始
- ルーム参加試行制限と共通エラー
- ゲーム状態取得
- 追加ドロー、カードプレイ、手番終了
- 収集カード選択、捨て札トップ選択
- 競合、権限不足、不正操作、終了済みゲームのエラー表現

### 画面設計

- 認証画面
- ルーム作成・参加画面
- 参加者待ち画面
- ゲーム画面
- ゲーム結果表示
