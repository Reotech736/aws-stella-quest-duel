# 09-ui-improvement

## 目的

Amplify Hostingで限定ベータ版を公開する前に、初見の利用者が現在のゲーム状況と次の操作を理解できるUIへ改善します。

重複収集ペナルティの宝石数を正しいルールへ修正した後、表示、操作フィードバック、レスポンシブ対応、演出を段階的に整えます。UI改善では、それ以外のゲームルールやサーバー状態遷移を変更しません。見栄えだけでなく、誤操作とルール上の情報漏えいを防ぐことを優先します。

## 現在の進捗

状態: 主要実装完了・手動回帰確認待ち

実装済み:

- 内部用語の利用者向け表示への変換
- 現在のフェーズ、次の操作、リードカラー、トランプカラーの表示
- 山札、捨て札、場、収集エリア、星明り、黒い星、スタートプレイヤーの盤面表示
- PCとスマートフォン縦向けのレスポンシブCSS
- ホームとゲーム画面から開けるルール説明
- 感情カード24種、休憩カード、裏面5種、星明り、黒い星のピクセルアート適用
- カード選択、場へのカード到着、収集、手札補充、黒い星、星明り反転の短い演出
- `prefers-reduced-motion`への対応
- Web Audio APIによる効果音、ミュート開始、音量設定のブラウザ保存

- 全フェーズ、追加ドロー、自動補充、投了、通常終了、再読込、ポーリング復帰の回帰確認
- 手動確認で見つかった致命的な重なりや操作不能の修正
- 確認結果を反映し、フェーズ9を完了へ更新する

## 着手時に確認した課題

- `OWNER`、`GUEST`、`DUMMY`などの内部表現が画面に残ることがある
- 山札、捨て札、収集エリア、黒い星、スタートプレイヤーの位置関係が直感的に分かりにくい
- カードが色と数字中心の仮表示で、カードゲームとしての視認性が低い
- カードを選択・確定したときの反応が弱い
- 黒い星やスタートプレイヤーの移動が状態の即時切り替えだけで表現される
- ボタンの形、文字、カード、固定操作領域が画面幅によって重なる可能性がある
- ログイン後にゲームルールや操作方法を確認する導線がない
- 自動補充などの自動処理は通知されるが、盤面の変化として理解しにくい
- サウンドがなく、手番変更やラウンド解決に気付きにくい

## UIで守るゲーム情報

演出を追加しても、次の公開範囲は変えません。

- 自分の手札は色、数字、カードイラストを表示する
- 相手の手札は枚数と裏面の色だけを表示し、数字や表面イラストを表示しない
- 山札は残り枚数と一番上のカードの裏面色だけを表示する
- 山札の上から2枚目以降の色、数字、順序は誰にも表示しない
- 場に出たカード、捨て札トップ、収集済みカードは表面を公開する
- アニメーション中にも、未公開情報を一瞬表示しない

## 優先度

P0とP1を限定ベータ公開前の対象とします。P0は操作性と理解に関わる必須品質、P1はカードゲームとしての体験品質です。P1を延期する場合は、Amplify作業へ進む前に項目単位で明示的に判断します。

### P0: 操作性と理解に関わる必須品質

#### 1. 用語とプレイヤー表示

- `OWNER`と`GUEST`は実際のプレイヤー名へ変換する
- `DUMMY`は「ダミー」と表示する
- 自分には「あなた」、相手には表示名を補助表示してもよいが、画面内で一貫させる
- APIの列挙値や内部識別子を、そのまま利用者向け文言にしない

#### 2. ゲーム状況と次の操作

- 現在のラウンド、手番プレイヤー、フェーズ、リードカラー、トランプカラーをまとめて表示する
- 「カードを選んでください」「獲得するカードを選んでください」など、次の操作を主文として表示する
- 操作できないカードと操作できるカードを見た目と`disabled`状態の両方で区別する
- 通信中、相手待ち、競合後の再読込、ゲーム終了を明確に区別する

#### 3. 盤面の視認性

- 山札をカードの束として表示し、残り枚数とトップの裏面色を併記する
- 捨て札トップを表向きで表示する
- 各プレイヤーの収集エリアを常時確認できるようにする
- 星明りトークンの光面・闇面を視覚的に区別する
- 黒い星がどのプレイヤーにあるか、または中央にあるかを表示する
- スタートプレイヤーと現在の手番を混同しない表示にする

#### 4. レスポンシブレイアウト

- PCとスマートフォン縦向きを正式確認対象とする
- ボタンの文字切れ、カードとの重なり、横スクロール、固定領域による盤面隠れをなくす
- 主要操作のタップ領域を十分に確保する
- 画面高さが小さい場合も投了と主要操作へ到達できるようにする

#### 5. ルール・操作ガイド

- ログイン後のホームとゲーム画面からルール説明を開けるようにする
- 最初は別画面を増やしすぎず、モーダルまたはドロワー形式を基本案とする
- 「勝利条件」「カードの強さ」「休憩カード」「星明りトークン」「画面操作」を短く分ける
- 詳細ルールは既存のルール文書と矛盾しないようにする

### P1: カードゲームとしての体験品質

#### 6. カードデザインとイラスト

- 感情カードは色、数字、宝石数、イラストを一体として読めるデザインにする
- 表面は4色×数字1〜6と休憩カードの計25種類とする
- 裏面は赤、黄、青、緑、休憩の計5種類とする
- 数字1、2には宝石3個、数字3、4には宝石2個、数字5、6には宝石1個を描く
- 休憩カードは感情カードと明確に区別する
- 各色のカード裏面と休憩カード裏面を用意する
- 既存ボードゲームの画像を権利確認なしに転用しない
- 現物写真は世界観と構図の参考に留め、直接トレースしないオリジナルのピクセルアートを共同制作する
- 完成した素材を`frontend/public`で管理し、参照写真はGitへ追加しない
- 画像が読み込めない場合も色、数字、代替テキストでゲームを続行できるようにする

素材仕様:

- カード全体・色別フレーム: 5:7、150×210px、PNG
- 背景込みイラスト: 2:3、120×180px、PNG
- イラストはカード内の`x=15, y=15`へ配置する
- 数字: 17×21px、透過PNGを四隅へ重ねる
- 宝石: 色別の12×15px透過PNGを上下の固定スロットへ重ねる
- 光・闇トークン: 256×256px、透過PNG
- 黒い星: 320×400px、透過PNG
- フレーム、数字、宝石も低解像度のドット絵で統一する
- ブラウザでは各PNGをレイヤー表示し、`image-rendering: pixelated`を適用する
- 宝石スロットは上下に左・中央・右の3枠を常設し、未使用枠を暗い空枠として残す
- 数字1・2は3枠、3・4は左右、5・6は中央へ宝石を表示する
- 画像が読み込めない場合は現行のHTML/CSSカード表示へフォールバックする

`workspace/tmp/photo`には表面25種、裏面5種、光・闇トークン、黒い星の参照写真が揃っています。感情カード24種、休憩カード、裏面5種は150×210pxの共通座標で作成済みです。同じ数字の4色は同一イラストを配色変換し、構図と輪郭が一致するように管理します。光・闇トークンと黒い星も透過ピクセルアートとしてゲーム画面へ適用済みです。

#### 7. 選択・確定エフェクト

- 選択中のカードを浮き上がり、枠、明度などで示す
- 選択とサーバー送信を分け、送信中の二重操作を防ぐ
- 成功時は場への移動、失敗時は元の位置への復帰として見せる
- 色だけに依存せず、枠線やアイコンも併用する

#### 8. マーカーとカード移動アニメーション

- 黒い星の移動、スタートプレイヤー変更、カードプレイ、カード獲得、自動補充を短いアニメーションで示す
- アニメーションはサーバーから受け取った確定状態を説明する表現とし、状態更新を遅らせない
- ポーリングで複数の変化をまとめて受信した場合は、長い連続演出にせず要点を通知する
- `prefers-reduced-motion`では移動量を減らすかアニメーションを無効化する

#### 9. サウンド

- カード選択、カード確定、手番開始、ラウンド解決、光を失う、ゲーム終了を候補とする
- 効果音はWeb Audio APIで短い8-bit風の音を生成する
- 初期状態はミュートとし、明示操作後だけ有効にしてブラウザの自動再生制限に従う
- SFXとBGMを別々にON/OFF・音量調整できる管理層を用意する
- BGMのループ再生基盤は実装するが、BGM音源の追加は今回の必須範囲外とする
- ヘッダーなどにサウンド設定を常設する
- 設定はブラウザ内に保存し、対戦相手へ共有しない
- 音がなくても同じ情報を画面から取得できるようにする
- 将来追加するBGM音源は利用条件を確認し、秘密情報や外部APIを必要としない静的ファイルを使う

### P2: 限定ベータ後でもよい改善

- 初回だけのチュートリアル
- 演出速度の個別設定
- テーマ切り替え
- 高度なカード移動やパーティクル表現
- 対局を振り返るための演出ログ

## 実装順序

1. 内部用語を画面表示から除去する。
2. ゲーム状況、次の操作、盤面領域を再構成する。
3. PCとスマートフォン縦向きの重なりを修正する。
4. ルール・操作ガイドを追加する。
5. カード、トークン、黒い星のピクセルアート試作を作り、画像管理方針を確定する。
6. 選択、カード移動、マーカー移動のアニメーションを追加する。
7. サウンドと設定UIを追加する。
8. 2ブラウザセッションでゲーム終了まで再確認する。

大きな一括変更にせず、各手順を独立してテスト・コミットします。CSSとブラウザ標準APIで実現できる範囲から始め、新しいUI、アニメーション、音声ライブラリは必要性を確認するまで追加しません。

## 確認方法

### 自動確認

- `npm run lint`
- `npm run typecheck`
- `npm test -- --run`
- `npm run build`
- 内部用語が利用者向け画面へ露出しないことのコンポーネントテスト
- フェーズごとの案内文と操作可否のコンポーネントテスト

### 手動確認

- PCの通常幅と狭い幅
- スマートフォン縦向き相当
- 通常ウィンドウとプライベートウィンドウによる2ユーザー対戦
- 全フェーズ、休憩カード、追加ドロー、自動補充、投了、通常終了
- サウンドOFF、モーション軽減設定
- ページ再読込とポーリングによる状態復元

ブラウザ自動操作の導入は別判断とし、まず既存のVitestと再現可能な手動チェックリストを充実させます。

- P1項目がすべて完了しているか、延期する項目が明示的に合意されている

- 相手の数字や山札2枚目以降などの非公開情報が表示されない
- どのフェーズでも現在の状況と次の操作を判断できる
- 内部用語が利用者向け画面に露出しない
- ルール説明へログイン後いつでも到達できる
- lint、typecheck、test、buildが成功する
