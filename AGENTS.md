# AGENTS.md

## プロジェクト概要

このリポジトリは、ボードゲーム「ステラクエスト」を二人対戦用Webアプリとして実装するためのプロジェクトです。

主な目的は、AWSの学習を兼ねて、小さく理解しやすいサーバーレスアプリケーションを作ることです。

## 基本方針

- まずは実装よりも設計・仕様整理を優先する。
- 最初から大規模な構成にしない。
- AWS学習目的のため、AWSのサーバーレス構成を優先する。
- MVPを小さく作り、後から機能追加する。
- 不要な依存ライブラリを勝手に追加しない。
- 実AWSリソースの作成・変更・削除は、明示的な確認なしに行わない。
- APIキー、トークン、認証情報、個人情報、社内情報をコミットしない。
- 判断に迷う場合は、実装前に方針を確認する。

## 初期アーキテクチャ方針

当面は、以下の構成を前提とする。

- フロントエンド: React または Next.js
- ホスティング: AWS Amplify Hosting
- 認証: Amazon Cognito
- API: Amazon API Gateway HTTP API
- バックエンド: AWS Lambda
- データベース: Amazon DynamoDB
- ログ: Amazon CloudWatch Logs

## ゲーム仕様の初期方針

- 対戦人数は2人固定とする。
- ターン制ゲームとして実装する。
- オンライン二人対戦のみを実装する。
- CPU対戦は現在も将来も実装対象外とする。
- 観戦、ランキング、チャットは後回しにする。
- 最初からWebSocketは使わず、HTTP API + ポーリングで始める。
- WebSocketやAppSync Subscriptionは、MVP完成後に必要性を再検討する。
- ゲームルールの判定は必ずサーバー側で行う。
- クライアント側は、表示と操作入力を主な責務とする。

## 状態管理方針

ゲーム状態は、DynamoDBに保存する。

初期方針として、以下の2種類のデータを扱う。

- 現在のゲーム状態
- 操作履歴、またはイベントログ

現在のゲーム状態には、以下のような情報を含める。

- gameId
- roomId
- プレイヤー情報
- 現在のターン
- 現在のフェーズ
- 手札
- 場に出ているカード
- スコア
- 勝敗
- 更新日時

イベントログには、以下のような情報を含める。

- gameId
- eventId
- userId
- actionType
- payload
- createdAt

## 開発順序

最初は以下の順番で進める。

1. `plan.md` を作成する。
2. MVP要件を整理する。
3. ゲームルールを文章化する。
4. DynamoDBのテーブル設計を整理する。
5. API一覧を整理する。
6. フロントエンドの画面構成を整理する。
7. 実装に入る。

## 最初に作るドキュメント

以下のファイルを優先して作成・更新する。

- `plan.md`
- `README.md`
- 必要に応じて `docs/architecture.md`
- 必要に応じて `docs/rules.md`

## Codexへの作業ルール

- いきなり実装を始めない。
- まず変更方針を説明する。
- 大きな変更を行う場合は、作業を小さな単位に分ける。
- 新しい依存関係を追加する場合は、理由を説明する。
- AWSリソースを作成するコマンドは、確認なしに実行しない。
- `.env` や認証情報を作成・編集する場合は、秘密情報を含めない。
- 生成したコードは、可能な範囲で動作確認方法も示す。
- 作業後は、変更したファイルと変更内容を要約する。

## 完了条件

以下を満たしたら完了と判断する。

- 関連ファイルが更新されている。
- 変更内容が説明されている。
- 必要に応じて実行方法・確認方法が書かれている。
- 不要な依存関係が増えていない。
- 秘密情報が含まれていない。
- Git差分を確認できる状態になっている。

## 禁止事項

- 明示的な確認なしにAWSへデプロイしない。
- 明示的な確認なしにAWSリソースを作成・削除しない。
- APIキー、アクセストークン、パスワードをコミットしない。
- 不要に複雑な構成へ変更しない。
- MVP前にランキング、チャット、観戦、課金などの追加機能を実装しない。
